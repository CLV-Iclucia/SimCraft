## Project

**SimCraft Python Bindings**

将 SimCraft C++ 物理模拟框架的核心 API 暴露为 Python 模块 `simcraft`，使用户可以用 Python 脚本（而非 JSON）配置和运行模拟系统。

**Core Value:** **用 Python 的动态逻辑替代静态 JSON 配置** — 用户获得条件分支、循环、变量、函数组合等编程能力来构建模拟场景，同时保持"配置完成后锁定系统"的安全模型。

## Naming Conventions
### Namespace Organization
- **Hierarchical namespaces** follow the module structure: `sim::<module>::<component>`
- All main project code resides in the `sim::` namespace
- External spatial data structures are in `spatify::` namespace
- Namespace braces: standard one line per level
### Type Names
- **Classes and Structs**: PascalCase with descriptive names
- **Template parameters**: PascalCase for types, UPPERCASE for constants
- **Type aliases**: snake_case with appropriate semantic meaning for STL-like, but remains PascalCase for most types
### Variable Names
- **Local variables and members**: snake_case
- **Constants**: UPPERCASE_SNAKE_CASE for compile-time constants
- **Enumerations**: PascalCase for enum names, UPPERCASE_SNAKE_CASE for values
### Function Names
- **Member functions**: camelCase with verb-noun structure
- **Factory methods**: PascalCase factory pattern
- **Operators**: Follow standard C++ conventions
### File Organization
- **Headers**: kebab-case with `.h` or `.hpp` extension
- **Implementation**: `.cc` extension (not `.cpp`)
- **Tests**: kebab-case with `-test.cc` or `test-*.cc` suffix
## Code Style and Formatting
### Formatting Standards
- **C++ Standard**: C++20 (enforced via `cmake_minimum_required(VERSION 3.20)` and `set(CMAKE_CXX_STANDARD 20)`)
- **Line length**: Implicit soft limit ~100 characters; harder limit ~120
- **Indentation**: 2 spaces (consistently applied across codebase)
- **Braces**: Allman style for classes/functions, K&R for inline code
- **Spacing**:
### Modern C++ Features Used
- **Auto type deduction**: Encouraged for complex types
- **Range-based for loops**: Preferred over index-based iteration
- **`std::optional`**: Used for nullable return values
- **`[[nodiscard]]`**: Applied to important return values
- **`noexcept`**: Explicitly marked for no-throw functions
### Comments and Documentation
#### Comment Density and Quality
- **File headers**: Auto-generated by CLion/IDE with date created and author
- **Header guards vs `#pragma once`**: Pragmatic mix
- **Class/struct documentation**: Provided inline with member comments
- **Function/method documentation**: Minimal; code clarity preferred
- **Inline comments**: Used sparingly for clarification
- **Commented-out code**: Generally avoided; use VCS history or TODO for future work
#### Comment Language
- **Primary**: English for all public interfaces and core algorithms
- **Chinese**: Used in test comments and internal documentation
### Language-Specific Style Notes
#### C++ Standard Library Usage
- **Containers**: `std::vector`, `std::array`, `std::unordered_map`
- **Memory management**: Smart pointers exclusively
- **Optional types**: `std::optional` for nullable values in APIs
- **String handling**: `std::string_view` for non-owning string parameters
- **Algorithms**: Use standard library algorithms with lambdas
#### GLM (Mathematics Library)
- **Vector/Matrix types**: GLM types preferred for 3D math
- **Integration**: GLM used in block operations; automatic conversion with Eigen via `.asEigen()`
#### Eigen Integration
- **Usage scope**: Primarily for sparse linear solvers and dense small-matrix operations
- **Boundary markers**: Explicit `.asEigen()` calls mark transitions to Eigen territory
## Architectural Conventions

- Lower-level modules are general-purpose utilities
- Higher-level modules use lower-level ones for domain-specific tasks
